# Portal Proiecte Web Marinciuc

Portal educațional pentru prezentarea, centralizarea și evaluarea peer-to-peer a website-urilor create de elevi în cadrul activităților de învățare HTML, CSS, publicare web și utilizare responsabilă a instrumentelor de inteligență artificială generativă.

Proiect coordonat de: **Tudor Lapp**

---

## 1. Descriere succintă a proiectului

**Portal Proiecte Web Marinciuc** este o platformă web statică, publicată prin GitHub Pages, care centralizează website-urile create de elevi și oferă un mecanism de evaluare peer-to-peer, adică evaluare realizată de colegi pe baza unor criterii clare, comune și transparente.

Proiectul are două scopuri principale:

1. **Scop didactic**: sprijinirea învățării HTML, CSS, organizării conținutului digital, publicării prin GitHub Pages și dezvoltării capacității elevilor de a construi produse digitale autentice.
2. **Scop formativ**: dezvoltarea competenței de a analiza, compara și evalua calitativ proiectele colegilor, oferind feedback argumentat, obiectiv și util.

Prin această platformă, elevii pot:

- să acceseze website-urile colegilor;
- să vizualizeze proiectele publicate;
- să evalueze proiectele folosind o grilă comună;
- să ofere comentarii constructive;
- să vadă scoruri agregate și sinteze generale;
- să înțeleagă mai bine ce înseamnă un website clar, util, relevant și bine realizat tehnic.

Platforma nu este doar o galerie de proiecte, ci un spațiu de învățare colaborativă. Fiecare proiect devine atât produs final, cât și resursă de învățare pentru ceilalți elevi.

### Ce competențe dezvoltă proiectul

Proiectul contribuie la dezvoltarea următoarelor competențe:

- utilizarea limbajului HTML pentru structurarea conținutului web;
- utilizarea CSS pentru stilizare și design vizual;
- înțelegerea principiilor de publicare online;
- utilizarea GitHub Pages pentru găzduirea unui website;
- folosirea responsabilă a instrumentelor GenAI și LLM;
- formularea de prompturi clare;
- verificarea și adaptarea codului generat de AI;
- analizarea critică a produselor digitale;
- oferirea de feedback constructiv;
- dezvoltarea creativității digitale;
- colaborarea și învățarea între colegi.

### Tehnologii utilizate

Proiectul folosește:

- **HTML** pentru structura paginii;
- **CSS** pentru stilizare și adaptare responsive;
- **JavaScript** pentru logica aplicației;
- **GitHub Pages** pentru publicarea proiectelor;
- **Google Sheets** pentru administrarea datelor;
- **Google Apps Script** pentru colectarea și agregarea evaluărilor;
- **GitHub Actions** pentru sincronizarea automată a datelor din Google Sheets către repository;
- **instrumente GenAI / LLM** pentru sprijin în generarea, explicarea, testarea și îmbunătățirea codului.

## 2. Instrucțiuni pentru elevi privind sarcina de lucru

Această secțiune explică ce are de făcut fiecare elev, de la etapa de concepere a website-ului până la publicarea, încărcarea linkului și evaluarea proiectelor colegilor.

### 2.1. Obiectivul sarcinii

Fiecare elev va crea un website propriu, publicabil online, folosind HTML, CSS și, acolo unde este necesar, JavaScript. Website-ul trebuie să aibă o temă clară, un public-țintă bine definit și o structură logică.

Website-ul poate avea una dintre următoarele forme:

- pagină de prezentare personală;
- website tematic;
- website de informare;
- website educațional;
- website pentru promovarea unei idei;
- website pentru un produs, serviciu sau proiect;
- mini-joc în browser;
- website de tip portofoliu;
- website cu instrucțiuni sau ghid practic;
- alt tip de proiect aprobat de profesor.

### 2.2. Ce înseamnă un website reușit

Un website reușit nu înseamnă doar o pagină frumoasă. El trebuie să fie clar, coerent, util și funcțional.

Un website bun trebuie să aibă:

- un titlu clar;
- o temă bine definită;
- conținut relevant;
- structură logică;
- secțiuni ușor de urmărit;
- design vizual coerent;
- culori și fonturi alese atent;
- imagini potrivite;
- linkuri funcționale;
- adaptare pentru ecrane diferite;
- cod cât mai curat și organizat;
- respectarea drepturilor de autor pentru imagini, texte și resurse.

### 2.3. Utilizarea instrumentelor GenAI și LLM

În cadrul acestei sarcini, elevii pot folosi instrumente de inteligență artificială generativă, inclusiv modele lingvistice mari, cunoscute și ca LLM.

**GenAI** înseamnă inteligență artificială generativă, adică sisteme care pot genera text, cod, imagini, explicații sau idei pornind de la instrucțiunile utilizatorului.

**LLM** înseamnă model lingvistic mare. Acesta este un sistem AI antrenat pe volume mari de text, capabil să genereze răspunsuri, explicații, exemple de cod și sugestii.

Elevii pot folosi GenAI pentru:

- generarea unei idei de website;
- formularea structurii paginii;
- generarea unui prim draft HTML;
- explicarea codului;
- corectarea erorilor;
- îmbunătățirea designului;
- generarea de exemple de conținut;
- verificarea accesibilității;
- adaptarea website-ului pentru telefon;
- înțelegerea unor concepte precum `div`, `section`, `header`, `footer`, `class`, `id`, `flexbox`, `grid`.

Totuși, AI-ul nu trebuie folosit pentru a înlocui complet munca elevului. Elevul trebuie să înțeleagă, să verifice și să poată explica website-ul creat.

### 2.4. Reguli pentru utilizarea AI

Elevii trebuie să respecte următoarele reguli:

1. Nu se copiază cod fără înțelegere.
2. Orice cod generat de AI trebuie testat.
3. Elevul trebuie să poată explica structura website-ului.
4. Elevul trebuie să personalizeze rezultatul.
5. Nu se acceptă proiecte complet generice, fără intervenție personală.
6. Conținutul trebuie adaptat temei și publicului-țintă.
7. Elevul trebuie să verifice corectitudinea informațiilor generate de AI.
8. Elevul trebuie să evite conținutul ofensator, incorect, copiat sau nepotrivit.
9. Imaginile și textele trebuie folosite responsabil.
10. Proiectul final trebuie să reflecte deciziile, creativitatea și munca elevului.

### 2.5. Etapele de lucru

#### Etapa 1. Alegerea temei

Elevul alege o temă pentru website.

Exemple de teme:

- CV digital;
- portofoliu personal;
- hobby personal;
- sport preferat;
- joc în browser;
- ghid de sănătate;
- pagină de prezentare a unei activități;
- proiect educațional;
- website despre o problemă socială;
- website despre o destinație turistică;
- website despre un produs imaginar;
- website despre protecția mediului;
- website despre siguranța online.

Tema trebuie să fie clară și suficient de bine delimitată pentru a putea fi prezentată într-un website.

#### Etapa 2. Stabilirea publicului-țintă

Elevul trebuie să decidă pentru cine creează website-ul.

Exemple de public-țintă:

- colegi de clasă;
- adolescenți;
- părinți;
- profesori;
- copii mai mici;
- pasionați de sport;
- persoane interesate de sănătate;
- utilizatori care caută informații practice;
- persoane interesate de jocuri;
- viitori angajatori, în cazul unui CV digital.

Publicul-țintă influențează limbajul, designul, structura și tipul de informație prezentată.

#### Etapa 3. Planificarea structurii

Înainte de a scrie codul, elevul trebuie să planifice structura website-ului.

O structură simplă poate include:

- antet;
- meniu;
- secțiune de introducere;
- secțiuni principale;
- imagini;
- butoane sau linkuri;
- footer;
- eventual o secțiune de contact sau concluzie.

Exemplu:

```txt
Header
├── Titlu website
├── Meniu de navigare

Main
├── Secțiunea 1: Prezentare
├── Secțiunea 2: Informații principale
├── Secțiunea 3: Galerie / exemple / resurse
├── Secțiunea 4: Concluzie

Footer
└── Autor, clasă, an
````

#### Etapa 4. Crearea website-ului

Elevul creează website-ul folosind:

* HTML pentru structură;
* CSS pentru design;
* JavaScript, dacă este necesar, pentru interactivitate.

Website-ul trebuie să conțină cel puțin:

* un titlu;
* mai multe secțiuni;
* text propriu sau adaptat;
* imagini relevante;
* design coerent;
* linkuri funcționale;
* footer;
* indicarea autorului;
* publicare prin GitHub Pages.

#### Etapa 5. Utilizarea GitHub

Elevul va încărca proiectul într-un repository GitHub.

Pașii generali sunt:

1. Crearea unui cont GitHub, dacă nu există deja.
2. Crearea unui repository nou.
3. Încărcarea fișierelor website-ului.
4. Activarea GitHub Pages.
5. Verificarea linkului public.
6. Testarea website-ului publicat.

Structura minimă recomandată a proiectului:

```txt
website-elev/
│
├── index.html
├── style.css
└── images/
```

sau, pentru proiecte mai simple:

```txt
website-elev/
│
├── index.html
└── style.css
```

Fișierul principal trebuie să se numească:

```txt
index.html
```

GitHub Pages îl folosește ca pagină principală a website-ului.

#### Etapa 6. Publicarea prin GitHub Pages

După încărcarea fișierelor în repository:

1. Elevul intră în `Settings`.
2. Selectează `Pages`.
3. Alege branch-ul `main`.
4. Selectează folderul `/root`.
5. Salvează.
6. Așteaptă publicarea linkului.
7. Deschide linkul și verifică dacă website-ul funcționează.

Linkul public va avea, de regulă, forma:

```txt
https://username.github.io/nume-repository/
```

#### Etapa 7. Încărcarea linkului pe Padlet

După publicarea website-ului, elevul trebuie să încarce linkul pe Padlet-ul clasei.

Postarea de pe Padlet trebuie să conțină:

* numele elevului;
* titlul proiectului;
* categoria proiectului;
* linkul GitHub Pages;
* o scurtă descriere a website-ului;
* eventual o captură de ecran.

Model de postare:

```txt
Nume: Prenume Nume
Titlul proiectului: Portofoliul meu digital
Categoria: CV / Portofoliu
Link: https://username.github.io/proiect/
Descriere: Website de prezentare personală, realizat în HTML și CSS, cu sprijinul instrumentelor GenAI pentru structurare și design.
```

#### Etapa 8. Verificarea apariției pe platformă

După ce linkul este colectat și introdus în sistem, proiectul va apărea pe platforma centralizată.

Elevul trebuie să verifice:

* dacă numele apare corect;
* dacă linkul se deschide;
* dacă proiectul este în categoria potrivită;
* dacă website-ul se vede corect pe telefon și calculator;
* dacă nu lipsesc imagini, stiluri sau fișiere.

#### Etapa 9. Vizualizarea proiectelor colegilor

Elevii vor accesa platforma și vor deschide proiectele colegilor.

Înainte de evaluare, elevul trebuie să exploreze website-ul colegului:

* să citească informațiile;
* să verifice meniul;
* să testeze linkurile;
* să observe designul;
* să analizeze structura;
* să verifice dacă website-ul este clar și ușor de folosit.

Evaluarea nu se face superficial. Elevul trebuie să vadă efectiv proiectul înainte de a completa formularul.

#### Etapa 10. Evaluarea peer-to-peer

Fiecare elev va evalua proiectele colegilor folosind formularul disponibil pe platformă.

Evaluarea trebuie să fie:

* obiectivă;
* respectuoasă;
* argumentată;
* bazată pe criterii;
* orientată spre îmbunătățire;
* utilă pentru autor.

Elevii nu trebuie să ofere note doar pe baza prieteniei, preferințelor personale sau aspectului general. Evaluarea trebuie să țină cont de criteriile stabilite.

### 2.6. Criterii de evaluare

Proiectele sunt evaluate pe baza următoarelor criterii:

| Criteriu                              | Ce se urmărește                                                                 |
| ------------------------------------- | ------------------------------------------------------------------------------- |
| Relevanța problemei abordate          | Website-ul are o temă clară, utilă sau interesantă.                             |
| Claritatea scopului și a mesajului    | Utilizatorul înțelege rapid despre ce este site-ul.                             |
| Adecvarea pentru publicul-țintă       | Limbajul, conținutul și designul sunt potrivite pentru publicul ales.           |
| Utilitatea și valoarea informațională | Informațiile sunt utile, relevante și bine selectate.                           |
| Structura și navigarea                | Site-ul este organizat logic și ușor de parcurs.                                |
| Design vizual și atractivitate        | Culorile, fonturile, imaginile și spațierea sunt folosite coerent.              |
| Calitatea tehnică a realizării        | Linkurile funcționează, pagina se încarcă bine, codul este funcțional.          |
| Originalitate și impact general       | Proiectul are o idee proprie și produce o impresie clară asupra utilizatorului. |

### 2.7. Cum se oferă feedback calitativ

Un feedback calitativ nu înseamnă doar „este frumos” sau „nu îmi place”. Feedback-ul trebuie să explice ce funcționează și ce poate fi îmbunătățit.

Exemple de feedback slab:

```txt
E frumos.
Nu-mi place.
Bravo.
Mai lucrează.
```

Exemple de feedback bun:

```txt
Website-ul are o structură clară, iar secțiunea de prezentare explică bine scopul proiectului. Ar putea fi îmbunătățită partea de navigare, deoarece unele butoane nu sunt suficient de vizibile.

Designul este atractiv și culorile sunt potrivite pentru tema aleasă. Totuși, pe telefon unele texte sunt prea apropiate, iar spațierea dintre secțiuni ar putea fi mărită.

Informația este utilă pentru publicul-țintă, dar ar fi bine să fie adăugate mai multe exemple sau imagini pentru a face conținutul mai convingător.
```

### 2.8. Reguli de conduită în evaluarea colegilor

Elevii trebuie să respecte următoarele principii:

1. Criticăm produsul, nu persoana.
2. Folosim un limbaj respectuos.
3. Oferim observații concrete.
4. Menționăm cel puțin un punct forte.
5. Propunem cel puțin o îmbunătățire.
6. Nu folosim ironii sau formulări ofensatoare.
7. Evaluăm obiectiv, conform criteriilor.
8. Nu oferim scoruri exagerat de mari sau mici fără motiv.
9. Scriem feedback care poate ajuta colegul.
10. Ne comportăm ca membri ai unei comunități de învățare.

### 2.9. Produsul final așteptat

La final, fiecare elev trebuie să aibă:

* un website funcțional;
* un link GitHub Pages valid;
* o postare pe Padlet-ul clasei;
* proiectul vizibil pe platforma centralizată;
* evaluări oferite colegilor;
* feedback primit de la colegi;
* eventual o versiune îmbunătățită a website-ului după feedback.

## 3. Detalii tehnice pentru profesori care vor să replice proiectul

Această secțiune este destinată profesorilor care vor să folosească același model pentru predarea creativă, colaborativă și aplicată a HTML-ului la liceu.

### 3.1. Ideea pedagogică

Proiectul pornește de la ideea că învățarea HTML devine mai relevantă atunci când elevii creează produse reale, publicate online, vizibile pentru colegi și evaluate prin feedback autentic.

În loc ca elevii să rezolve doar exerciții izolate de tip „scrie un tag” sau „creează o listă”, ei construiesc un website complet, cu scop, public și utilizatori reali.

Acest model combină:

* învățarea prin proiect;
* învățarea colaborativă;
* evaluarea peer-to-peer;
* publicarea autentică;
* utilizarea responsabilă a inteligenței artificiale;
* dezvoltarea competențelor digitale;
* reflecția asupra calității produsului digital.

### 3.2. Arhitectura tehnică a proiectului

Proiectul este organizat ca website static, cu date externe și integrare Google.

Structura tehnică generală:

```txt
GitHub Pages website
│
├── index.html
├── styles.css
├── scripts.js
├── elevi.json
│
├── Google Sheets pentru lista proiectelor
├── GitHub Actions pentru sincronizarea elevi.json
│
└── Google Apps Script + Google Sheets pentru evaluări
```

Rolul componentelor:

| Componentă         | Rol                                                                           |
| ------------------ | ----------------------------------------------------------------------------- |
| `index.html`       | Structura paginii web.                                                        |
| `styles.css`       | Designul vizual și adaptarea responsive.                                      |
| `scripts.js`       | Logica aplicației: încărcare proiecte, afișare carduri, evaluare, comentarii. |
| `elevi.json`       | Lista proiectelor afișate pe platformă.                                       |
| Google Sheets      | Interfață simplă de administrare pentru profesor.                             |
| GitHub Actions     | Automatizează actualizarea `elevi.json` din Google Sheets.                    |
| Google Apps Script | Primește evaluările și returnează scorurile agregate.                         |
| GitHub Pages       | Publică platforma online.                                                     |

### 3.3. Fluxul datelor pentru proiecte

Lista proiectelor este administrată într-un Google Spreadsheet.

Profesorul poate gestiona proiectele într-un tabel cu următoarele coloane:

```txt
id | nume | url | categorie | activ
```

Câmpurile au următorul rol:

| Câmp        | Explicație                                                   |
| ----------- | ------------------------------------------------------------ |
| `id`        | Identificator unic al proiectului. Trebuie să rămână stabil. |
| `nume`      | Numele elevului sau al echipei.                              |
| `url`       | Linkul GitHub Pages al proiectului.                          |
| `categorie` | Categoria proiectului.                                       |
| `activ`     | Controlează dacă proiectul apare sau nu pe site.             |

Exemplu:

```txt
id              | nume             | url                                                | categorie | activ
mihaela-urzica  | Mihaela Urzica   | https://username.github.io/proiect/               | CV        | TRUE
balaban-inna    | Inna Balaban     | https://username.github.io/proiect/               | CV        | TRUE
demo-proiect    | Proiect Demo     |                                                    | demo      | FALSE
```

### 3.4. De ce este important câmpul `id`

Câmpul `id` este cheia tehnică a proiectului. Acesta conectează proiectul afișat pe platformă cu evaluările salvate în Google Sheets.

Dacă un proiect are:

```txt
id = balaban-inna
```

atunci evaluările pentru acest proiect vor avea:

```txt
projectId = balaban-inna
```

Dacă profesorul schimbă ulterior `id`, evaluările deja colectate nu se vor mai conecta corect cu proiectul.

Recomandare:

```txt
prenume-nume
```

Exemple bune:

```txt
mihaela-urzica
balaban-inna
felicia-odajiu
```

De evitat:

```txt
Mihaela Urzica
-elev
test!!!
```

### 3.5. Sincronizarea proiectelor din Google Sheets

Pentru a evita modificarea manuală a fișierului `elevi.json`, proiectul poate folosi un workflow GitHub Actions.

Fluxul este:

```txt
Profesorul modifică Google Sheet
        ↓
Google Sheet este publicat ca CSV
        ↓
GitHub Actions citește CSV-ul
        ↓
Scriptul generează elevi.json
        ↓
GitHub face commit automat
        ↓
GitHub Pages actualizează site-ul
```

Avantajul acestei soluții este că profesorul nu trebuie să modifice codul site-ului de fiecare dată când apare un proiect nou.

### 3.6. Publicarea Google Sheet-ului ca CSV

Pentru ca GitHub Actions să citească datele, Google Sheet-ul trebuie publicat ca CSV.

Pași:

1. Deschide Google Spreadsheet.
2. Selectează foaia cu proiectele.
3. Accesează `File`.
4. Selectează `Share`.
5. Selectează `Publish to web`.
6. Alege foaia corectă.
7. Alege formatul `Comma-separated values (.csv)`.
8. Copiază linkul publicat.

Linkul va avea o formă asemănătoare:

```txt
https://docs.google.com/spreadsheets/d/e/.../pub?gid=...&single=true&output=csv
```

Acest link este introdus în GitHub ca secret, de exemplu:

```txt
SHEET_CSV_URL
```

### 3.7. Actualizarea ordinii proiectelor

Dacă profesorul vrea să controleze ordinea proiectelor, există două opțiuni.

Prima opțiune este păstrarea ordinii rândurilor din Google Sheets. În acest caz, scriptul care generează `elevi.json` nu trebuie să sorteze alfabetic datele.

A doua opțiune, mai recomandată, este adăugarea unei coloane:

```txt
ordine
```

Exemplu:

```txt
ordine | id              | nume             | url | categorie | activ
1      | tudor-lapp      | Tudor LAPP       | ... | BOARD     | TRUE
2      | mihaela-urzica  | Mihaela Urzica   | ... | CV        | TRUE
3      | balaban-inna    | Inna Balaban     | ... | CV        | TRUE
```

Această coloană permite profesorului să controleze explicit ordinea de afișare fără să depindă doar de mutarea rândurilor.

### 3.8. Fluxul evaluărilor

Evaluarea proiectelor se realizează printr-un formular integrat în platformă.

Când elevul trimite o evaluare:

```txt
Formular din platformă
        ↓
Google Apps Script
        ↓
Google Sheets
        ↓
Sinteză agregată
        ↓
Afișare pe platformă
```

Evaluarea conține:

* `projectId`;
* numele proiectului;
* categoria proiectului;
* linkul proiectului;
* numele evaluatorului;
* clasa evaluatorului;
* confirmarea că site-ul a fost vizitat;
* scoruri pentru criterii;
* comentarii formative;
* data trimiterii.

### 3.9. Criteriile de evaluare folosite în platformă

Platforma folosește opt criterii, fiecare cu pondere proprie:

| Criteriu                              | Pondere |
| ------------------------------------- | ------: |
| Relevanța problemei abordate          |     15% |
| Claritatea scopului și a mesajului    |     10% |
| Adecvarea pentru publicul-țintă       |     15% |
| Utilitatea și valoarea informațională |     15% |
| Structura și navigarea                |     10% |
| Design vizual și atractivitate        |     10% |
| Calitatea tehnică a realizării        |     10% |
| Originalitate și impact general       |     15% |

Scorul general este calculat ca medie ponderată.

### 3.10. Comentariile formative

Pe lângă scorurile numerice, elevii oferă comentarii în patru câmpuri:

| Câmp                 | Rol                                            |
| -------------------- | ---------------------------------------------- |
| `strengthComment`    | Ce este reușit în proiect.                     |
| `improvementComment` | Ce trebuie îmbunătățit.                        |
| `audienceComment`    | Cât de bine răspunde site-ul publicului-țintă. |
| `generalComment`     | Comentariu general pentru autor.               |

Aceste comentarii pot fi afișate în platformă prin fereastra de tip pop-up „Vezi comentariile”.

### 3.11. Recomandări pentru configurarea Google Apps Script

Google Apps Script trebuie să aibă două funcționalități principale:

1. `doPost(e)` pentru salvarea evaluărilor trimise din website.
2. `doGet(e)` pentru returnarea sintezelor agregate către website.

Răspunsul trimis către website trebuie să includă:

```json
{
  "ok": true,
  "projects": {
    "project-id": {
      "count": 3,
      "weightedScore": 4.2,
      "criterionAverages": {},
      "scoreLabel": "Foarte bun",
      "summary": "Text de sinteză",
      "comments": []
    }
  }
}
```

Câmpul `comments` este important dacă profesorul dorește afișarea feedbackului individual.

### 3.12. Publicarea Apps Script

Pentru ca platforma să poată trimite și primi date, Apps Script trebuie publicat ca Web App.

Setări recomandate:

```txt
Execute as: Me
Who has access: Anyone
```

După orice modificare în Apps Script, trebuie făcut un nou deployment:

```txt
Deploy
→ Manage deployments
→ Edit
→ Version: New version
→ Deploy
```

Doar salvarea codului nu actualizează automat Web App-ul public.

### 3.14. Cum poate un profesor să reproducă proiectul

Un alt profesor poate replica proiectul parcurgând următorii pași:

1. Face fork sau copie a repository-ului.
2. Activează GitHub Pages.
3. Creează un Google Sheet cu lista proiectelor.
4. Publică Google Sheet-ul ca CSV.
5. Configurează secretul `SHEET_CSV_URL` în GitHub.
6. Rulează workflow-ul de sincronizare.
7. Creează un Google Sheet pentru evaluări.
8. Creează și publică Google Apps Script ca Web App.
9. Introduce URL-ul Apps Script în configurația JavaScript.
10. Testează încărcarea proiectelor și trimiterea evaluărilor.
11. Oferă elevilor instrucțiunile de lucru.
12. Colectează linkurile proiectelor prin Padlet sau Google Form.
13. Publică proiectele pe platformă.
14. Organizează sesiunea de evaluare peer-to-peer.
15. Analizează rezultatele și feedbackul cu elevii.
